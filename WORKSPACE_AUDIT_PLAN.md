#### 5. Validate Apple Best Practices Compliance

**Architecture Validation**:

**BaseViewModel MVVM Pattern**:
- All ViewModels must conform to `BaseViewModel` protocol from `Shared/Sources/SharedKit/SharedArchitecture.swift`
- Properties: `state`, `isLoading`, `errorMessage`
- Methods: `handle(_ action: Action) async`, `resetError()`, `validateState()`
- Use `@MainActor` for UI-bound ViewModels
- Verify across all 5 projects

**Platform-Specific Code**:
- Proper use of `#if os(iOS)` / `#if os(macOS)` / `#if os(iPadOS)` conditionals
- No iOS-specific APIs in macOS builds (e.g., `navigationBarTitleDisplayMode`)
- Platform-appropriate UI patterns (TabView for iOS, NavigationSplitView for macOS)

**Cross-Platform Toolbar Placements**:
- Use `.primaryAction` instead of `.navigationBarTrailing`
- Use `.cancellationAction` instead of `.navigationBarLeading`
- Use `.automatic` for flexible placement

**Data Model Separation** (CRITICAL):
- **Zero SwiftUI imports in `SharedTypes/` folder**
- Pure data models only (structs, enums, no UI logic)
- UI extensions in `Extensions/` folder only
- Prevents circular dependencies and concurrency issues

**Thread Safety**:
- Adopt `Sendable` protocol for types crossing concurrency boundaries
- Use `@MainActor` for UI-bound classes
- Background processing on `DispatchQueue.global()`
- MainActor updates via `Task { @MainActor in ... }`

**Avoid Codable in Complex Models**:
- No `Codable` conformance in nested, complex types
- Use separate DTO (Data Transfer Object) types for serialization
- Prevents circular dependencies and compiler errors

**Naming Conventions**:
- Avoid generic names: "Manager", "Dashboard", "Handler"
- Use specific descriptive names: "ProjectAnalysisCoordinator", "PerformanceMetricsDashboard"
- Prevents naming conflicts as codebase grows
